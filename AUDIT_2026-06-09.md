## Threat model (calibrated)

Single-user Windows PyQt6 desktop app. No server, no multi-tenant input. The realistic adversaries are:
1. **Compromised / MITM'd upstream scrape sources** (Zacks, Finviz, Finnhub, Yahoo, Nasdaq, SEC) returning hostile responses;
2. **Local file tampering / corruption** of the "sacred" caches (OHLCV parquets, `earnings_history.parquet`, `earnings_dates.parquet`, `universe.csv`);
3. **Misdirected automation** — `pyautogui` typing into the wrong window.

Security severities are scaled to this model; issues requiring the user to attack their own machine are not inflated.

## The four themes that matter most

1. **Cache-corruption races (data integrity).** The `HISTORY_WRITE_LOCK` that protects `earnings_history.parquet` is bypassed on the `_finalize_fill` and Auto-fix paths, and `earnings_dates.parquet` has *no* lock at all — while the launch-time smart refresh runs finviz/zacks (optionally finnhub) workers **concurrently**. Plus `atomic_write_parquet` uses a *fixed* temp-file name, so two concurrent writers can clobber each other's `.tmp` and produce a genuinely corrupt file. This cluster (H2, H3, M-load-cache-guard, M-incremental-truncate) is the highest-value fix: it silently destroys multi-hour-scraped caches, and the fixes are **low** regression risk.

2. **Misdirected `pyautogui` automation (safety).** The TradeStation bridge and per-row hotkey type ticker symbols + Enter/Tab into whatever window has focus, with no target-window verification (H5), no `try/except` around the bridge worker so a failsafe abort permanently locks the UI buttons (H4), no countdown/coordinate-validation on the hotkey (M-hotkey-countdown), and an unbounded concurrent-dispatch race on shared `pyautogui` global state (M-hotkey-race). With a brokerage order ticket focused, this can mistype into live order fields.

3. **GUI main-thread blocking (usability).** A dozen actions run network fetches or full-parquet read/rewrites synchronously on the GUI thread — spot fills (up to ~30 s), rebuild-tickers, reconcile, integrity auto-fix, coverage dialogs, startup universe load. The window goes "Not Responding"; a user may force-kill mid-write (which feeds back into theme 1).

4. **Confidently-wrong output (correctness).** CSV export writes static headers over dynamically-reordered columns, so labels don't match the data beneath them (H6). An enabled RS-S&P/NASDAQ filter silently passes *all* tickers when the benchmark column is missing (M-rs-filter). Calendar-only `last_earnings` goes stale a quarter behind after a report (M-stale-last).

### H1. compute_yoy_columns is an O(n) Python row loop over the full ~138k-row parquet on every scan setup and fill finalize `[efficiency]`
- **ID:** `eff-compute-yoy-python-row-loop`  |  **Fix regression risk:** medium
- **Location:** trade_scanner_fh/earnings_history.py:555-601 (loop body); callers: scanner.py:1522, earnings_history.py:1330, finnhub_fill.py:464, finviz_fill.py:410 (plus spot-fill finalize at finnhub_fill.py:852 / finviz_fill.py:766)
- **What:** compute_yoy_columns iterates every row in pure Python: an outer groupby('ticker') then an inner for i in idx that constructs cur_p - pd.DateOffset(years=1) per row and does scalar df.at[i,col] / df.at[prior_i,col] lookups (4 per row) plus float conversions. DateOffset construction and .at scalar access are slow; over ~138k rows this is hundreds of thousands of Python ops. It runs at scanner.py L1522 on every scan kickoff (after global dedup) and again in each fill worker's _finalize_fill — up to 3x per concurrent smart refresh. A secondary correctness note: the prior-year resolution does 'last wins' positionally, not the updated_at-latest the docstring claims (L560-565), which can pick the wrong same-slot source on a pre-dedup frame.
- **Impact:** Adds avoidable seconds of single-threaded CPU to every scan start (on the scan-setup path before results appear) and every fill finalize. On a pre-dedup frame the positional 'last wins' can also compute yoy against the wrong source's prior-year value.
- **Recommended fix:** Vectorize. Build a per-row prior-year key once as a vector: `prior_key = period_ts - pd.DateOffset(years=1)` (preserves the exact leap-year/day-1 semantics the current code relies on — do NOT substitute a fixed 365-day Timedelta). Then self-merge the frame onto itself on `["ticker", merged-period]` to attach each row's prior-year reported_eps/reported_rev as `prior_eps`/`prior_rev`, and compute `yoy = (cur - prior)/prior.abs()*100` with a vector mask `prior.abs() < MIN_YOY_*_BASE` set to NaN (reproducing the L595-596 denominator floor and the L583 NaN-skip). This collapses ~14 s to well under a second. To keep results bit-identical, on a frame that already has one row per (ticker, period_ending) the merge is unambiguous; if you want to be defensive against any transient same-slot dups in the finalize paths, drop_duplicates / sort by `updated_at` DESC before the merge so the "prior" side deterministically picks the latest update (this ALSO fixes the L557-565 comment/code mismatch). Lower-effort partial win with near-zero regression risk: only call compute_yoy_columns on a deduped frame, and skip the redundant scanner.py L1522 recompute when the on-disk yoy columns are already current (the parquet already stores yoy_*_pct), or move the heavy recompute out of the synchronous scan-setup path. Whichever route, add a unit test asserting the vectorized output equals the current loop output on a fixture covering negative-prior, near-zero-base (floor), missing-prior, and leap-year-Feb cases before swapping.

### H2. Fill _finalize_fill performs an unlocked read-modify-write of earnings_history.parquet, racing concurrent fills `[robustness]`
- **ID:** `robust-finalize-history-unlocked-write`  |  **Fix regression risk:** low
- **Location:** trade_scanner_fh/finviz_fill.py:403-411 and trade_scanner_fh/earnings_history.py:1313-1335 (both `_finalize_fill`); trade_scanner_fh/finnhub_fill.py:453+ (the finnhub `_finalize_fill` has the identical unlocked pattern, called at L728/L852 — reporter did not cite it but it is the same defect); trade_scanner_fh/gui/main_window.py:4833-4841 (modeless Auto-fix `_do_fix`, with df captured at L4776). save_earnings_history itself: earnings_history.py:233-306 (does not lock internally).
- **What:** main_window._launch_smart_refresh_workers() starts the finviz and zacks (optionally finnhub) fill workers CONCURRENTLY against one candidate list. The per-flush write path serializes via HISTORY_WRITE_LOCK, but each source's end-of-fill _finalize_fill() does load_earnings_history() → compute_yoy_columns() → save_earnings_history() WITHOUT acquiring HISTORY_WRITE_LOCK (finviz_fill.py L408-411; earnings_history.py _finalize_fill L1325-1331; the integrity auto-fix path in main_window.py L4833-4841 likewise overwrites via save_earnings_history without the lock or a running-worker guard). Because fills finish near the same time, one source's unlocked finalize can interleave with another's locked flush: it reads a stale snapshot, the other writes new rows, then the stale snapshot is written back — silently dropping the just-written rows.
- **Impact:** Lost-update corruption of earnings_history.parquet (the user's sacred multi-hour-scraped cache) during a concurrent launch-time smart refresh, or when the modeless Auto-fix dialog overwrites the file while a fill worker is appending. Freshly-scraped quarters can be silently discarded; the loss is invisible until a later scan shows a stale quarter. This is exactly the failure HISTORY_WRITE_LOCK was added to prevent, reintroduced on the finalize/auto-fix paths.
- **Recommended fix:** Wrap only the load→compute_yoy→save sequence inside each `_finalize_fill` in `with eh.HISTORY_WRITE_LOCK:` — i.e. finviz_fill.py L408-411, earnings_history.py L1325-1331, and (for completeness) finnhub_fill.py's `_finalize_fill`. Keep the subsequent `reconcile_earnings_dates(...)` call OUTSIDE the lock so the IO-heavy reconcile (which has its own save/lock paths) is not serialized under the history lock — wrapping only the read+write closes the race without over-serializing. The lock is an RLock, so this nests safely if any path re-enters.

For the Auto-fix path (main_window.py `_do_fix`): (a) acquire the lock and RE-READ inside the critical section rather than using the stale `df` captured at dialog-open — `with eh.HISTORY_WRITE_LOCK: df_now = load_earnings_history(); fixed, msgs = fix_integrity_issues(df_now, findings); save_earnings_history(fixed, sort=True)`; and (b) refuse the fix (or warn) when `self._earn_threads_active()` returns True (that helper already exists — used at main_window.py L1940) so a mid-flight fill can't be clobbered. Re-reading inside the lock is the robust part; the running-worker guard is belt-and-suspenders.

A simpler alternative if minimizing edits: make `save_earnings_history` acquire `HISTORY_WRITE_LOCK` internally — but that does NOT fix the lost-update, because the read (`load_earnings_history`) still happens outside the lock. The fix must wrap the read+compute+write together, so per-callsite wrapping is the correct approach.

### H3. earnings_dates.parquet read-modify-write is unlocked while finviz/zacks/yahoo/nasdaq fills run concurrently (lost updates + tmp-collision corruption) `[robustness]`
- **ID:** `robust-earnings-dates-parquet-unlocked-rmw`  |  **Fix regression risk:** low
- **Location:** trade_scanner_fh/earnings_cache.py:103-122 (_merge_and_save, save_earnings_cache:75-79); trade_scanner_fh/earnings_reconcile.py:285-375 (unlocked load→save); trade_scanner_fh/config.py:77-82 (fixed-name tmp in atomic_write_parquet); concurrency source trade_scanner_fh/gui/main_window.py:1879-1907; finviz mid-loop flush trade_scanner_fh/finviz_fill.py:396 + 491-505; reconcile callers gui/workers.py:826, finviz_fill.py:632/641; locked-sibling contrast earnings_history.py:89 + 1282
- **What:** earnings_dates.parquet is written via load→modify→save in at least four places (nasdaq_fill, yahoo_fill targeted/spot, finviz_fill._flush_next_dates_to_cache, earnings_reconcile.reconcile_earnings_dates). Unlike earnings_history.parquet (serialized by HISTORY_WRITE_LOCK), the dates cache has NO shared lock. The smart-refresh path starts the finviz and zacks workers concurrently; both call reconcile_earnings_dates near end-of-run, and finviz additionally flushes to the cache periodically. Whichever writer reads first and saves last silently discards the other's rows (lost update). Compounding this, atomic_write_parquet (config.py L77-82) derives a FIXED temp filename (path.suffix + '.tmp'), so two concurrent writers open the identical tmp; their pyarrow writes can interleave and one os.replace can promote a tmp the other was mid-writing — actual file corruption, not just lost updates. A corrupt read then returns None and re-floods smart refresh.
- **Impact:** During an overnight smart refresh (or any overlapping yahoo/nasdaq fill), one source's freshly-written last/next earnings dates can be silently dropped, leaving stale/missing next_earnings; in the worst case the parquet is left truncated/corrupt so loaders return None and the Days-Since/Days-Until ER filters go blank. Wrong/empty earnings calendar shown as authoritative until the next clean fill.
- **Recommended fix:** Two independent, low-risk fixes (apply both):

1) Add a module-level lock in earnings_cache.py: `DATES_WRITE_LOCK = threading.RLock()` (re-entrant, matching HISTORY_WRITE_LOCK). Wrap the full load→merge→save critical section: inside `_merge_and_save` acquire it around the concat/drop_duplicates/save_earnings_cache block, AND in earnings_reconcile.reconcile_earnings_dates acquire the SAME lock around `dates_df = ec.load_earnings_cache()` ... `ec.save_earnings_cache(combined)`. To keep load-vs-save consistent, the reconcile must load dates_df INSIDE the lock (currently it can be loaded at L288 before the merge) — i.e. acquire the lock spanning the dates_df load through the save when dates_df was not passed in by a caller/test. Expose the lock via earnings_cache so reconcile imports the one true lock (avoid a second lock object). Re-entrancy means finviz's _finalize_fill (reconcile) after a flush won't self-deadlock on the same thread.

2) Make atomic_write_parquet (and atomic_write_text) use a unique tmp name so unsynchronized writers can never share/clobber the same .tmp: `tmp = path.parent / f'.{path.name}.{os.getpid()}.{uuid.uuid4().hex}.tmp'` then `os.replace(tmp, path)` (os.replace is atomic on the same volume on Windows; both files are in scanner_data so same volume). This also future-proofs any other parquet writer. Add a `finally` to unlink the tmp if to_parquet raises, so unique tmps don't accumulate.

Note the lock only serializes writers within ONE process; that is sufficient here because all four writers run as QThreads in the single GUI process. The RAW_RETENTION_DAYS cleanup won't touch .tmp files (different suffix), so leftover unique tmps from a crash are harmless but should be unlinked on failure.

### H4. BridgeWorker.run() has no try/except — a pyautogui FailSafeException permanently locks Send/Scan/Manual buttons `[robustness]`
- **ID:** `robust-bridgeworker-no-try-except`  |  **Fix regression risk:** low
- **Location:** trade_scanner_fh/gui/workers.py:1085-1086 (BridgeWorker.run); contributing: trade_scanner_fh/tradestation.py:95-99 (start finally restores globals only, no on_done) and 197-198 (on_done unreachable on exception); GUI lock surfaces at trade_scanner_fh/gui/main_window.py:6938-6945 (only `done` connected, no `finished` fallback) and 7059-7072 (_on_bridge_done is the sole re-enable path).
- **What:** BridgeWorker.run() calls self.bridge.start() with no try/except and never emits its done signal itself. TradeStationBridge.start() runs pyautogui.typewrite/press with FAILSAFE enabled (tradestation.py:93). If the user slams the mouse into a screen corner (the documented abort gesture) or any pyautogui call raises, FailSafeException propagates out of start() and escapes run(), so the bridge's normal on_done(...) call (tradestation.py:197-198) is never reached and the GUI's _on_bridge_done slot never fires.
- **Impact:** The buttons disabled at launch — btn_scan, btn_send, btn_manual — stay disabled forever and btn_stop_send stays enabled, with _bridge_worker never reset to None; the user cannot scan/send/manually enter tickers again without restarting the app. Additionally, an uncaught exception escaping QThread.run() under PyInstaller windowed mode can abort the whole process (the exact failure ScanWorker guards against at workers.py:100-107). The failsafe corner-move is the intended user abort mechanism, so this path is likely.
- **Recommended fix:** Wrap BridgeWorker.run() body in try/except mirroring ScanWorker (workers.py:100-107): `try: self.bridge.start() except Exception as exc: log.error('BridgeWorker crashed: %s', exc, exc_info=True); self.log_msg.emit(f'Send error: {exc}'); self.done.emit(0, 0)`. Emitting done(0,0) routes through _on_bridge_done, which re-enables btn_scan/btn_send/btn_manual, disables btn_stop_send, and clears _bridge_worker. This is safe: _show_stop_sending_options only fires when _stop_sending_show_dialog is True (main_window.py:7077), which is set only by an explicit Stop-Sending click, so a failsafe abort will NOT spuriously pop the Resume/Restart dialog. Optionally catch pyautogui.FailSafeException first with a friendlier 'Send aborted (failsafe corner-move)' message before the broad Exception. Do not add a second `finished`-signal re-enable connection as well — that would double-handle and could race the `done` emit.

### H5. pyautogui types into whatever window has focus — no target-window verification (bridge + hotkey) `[robustness]`
- **ID:** `robust-pyautogui-no-target-window-verification`  |  **Fix regression risk:** medium
- **Location:** trade_scanner_fh/tradestation.py:133-143 (bridge typewrite/press loop) and trade_scanner_fh/hotkey.py:169-184 (send_ticker click/typewrite/end-key); HotkeyConfig.normalized() coordinate-validation gap at hotkey.py:105-120; GUI dispatch with no guard at gui/main_window.py:4308-4332 and 6920-6945. Cited 138-143 is accurate (the typewrite/press calls are at 142-143).
- **What:** The watchlist bridge and the per-row hotkey (hotkey.py) drive raw pyautogui keystrokes/clicks with no check that the intended target application/window actually has keyboard focus (no GetForegroundWindow/getActiveWindow gating exists in production). The bridge relies entirely on the user clicking the correct field during a countdown; the hotkey relies on a saved screen coordinate being valid. If the user alt-tabs, a notification steals focus, the target window moves, or the saved hotkey coordinate is stale (resolution/monitor/layout change), the loop blindly types ticker strings + presses Enter/Tab into the wrong window.
- **Impact:** Ticker symbols and Enter/Tab/Ctrl+Enter keystrokes get injected into whatever app has focus — a brokerage order ticket, chat, terminal, or document. With a brokerage front-end focused this can mistype into live order fields. The batch loop can send dozens of tickers + Enter presses before the user reacts.
- **Recommended fix:** Add an opt-in target-window guard using the already-bundled pywin32 (win32gui is imported in zacks_scraper.py, so the dependency is present). Lowest-regression approach: (1) When the user sets up the bridge/hotkey, capture the foreground window once (win32gui.GetForegroundWindow + GetWindowText) and store the title substring as an OPTIONAL expected-target preference. (2) Before each batch send (and before each hotkey fire), if an expected target is set, compare win32gui.GetWindowText(GetForegroundWindow()) against it; if it doesn't match, abort with a clear log message instead of typing. Keep this OFF by default / best-effort so headless tests and non-Windows paths are unaffected (the modules already import pyautogui lazily and inject a test seam). (3) In HotkeyConfig.normalized(), validate click_x/click_y (and return_click_x/y) against the current virtual-desktop bounds (pyautogui.size() or win32api.GetSystemMetrics SM_CXVIRTUALSCREEN/SM_CYVIRTUALSCREEN); if out of range, treat has_position as False so send_ticker refuses to fire a stale coordinate (it already returns False with a helpful message when has_position is False, lines 156-159 — reuse that path). All guards should be additive and fail-open only when no expected target is configured, so existing behavior is preserved for users who don't opt in.

### H6. CSV Export writes static RESULT_COLUMNS headers over the dynamic/reordered model columns `[robustness]`
- **ID:** `gui-csv-export-wrong-headers`  |  **Fix regression risk:** low
- **Location:** trade_scanner_fh/gui/main_window.py:6831-6845 (header build 6833-6836; data write 6840-6845). Trigger wiring at main_window.py:1170-1178. Root mechanism in trade_scanner_fh/gui/widgets.py:1369-1398 (_build_dynamic_columns filter+append) and 2402-2409 (populate assigns model columns from cols).
- **What:** _export_results reads row data from the live proxy model (which after populate() holds the dynamic column set from _build_dynamic_columns — populated-only columns plus q-i beats blocks, in the user's reordered/hidden order) but builds the CSV header row from the static module constant RESULT_COLUMNS indexed positionally. The model's column set almost never matches RESULT_COLUMNS: dynamic columns drop unpopulated keys and append q{1..N}_* beats columns, so headers are misaligned with data and, when beats columns push the count past len(RESULT_COLUMNS), trailing data columns get no header at all.
- **Impact:** The CSV Export path emits a file whose column labels don't correspond to the values beneath them — wrong/mislabeled trading data presented as a correct export. Triggers on essentially every real scan (any disabled indicator or beats data changes the active column set). The Excel button uses the correct active_columns path, which is the only mitigation.
- **Recommended fix:** In the CSV branch of `_export_results`, derive both the header labels and the iterated column count from the live model layout instead of the static RESULT_COLUMNS. Concretely, read `active = self.results_table.active_columns` (the same list the model was built from in `_populate_impl`) and write `headers = [h for h, _k, _f in active]`, then iterate data over `for c in range(len(active))` (which equals `proxy.columnCount()`). This guarantees one header per data column, in the model's actual order, including dropped-column compaction and appended Q-i beats columns. Lowest-risk variant: simply replace the header list comprehension with `headers = [h for h, _k, _f in self.results_table.active_columns]` and leave the data loop as-is (it already iterates the full `cols`=proxy.columnCount()); the two counts will then match. Note `active_columns` is the canonical order, not the user's drag-reordered VISUAL order — to also honor visual reordering, route this button through the existing `_ordered_active_columns_for_export()` / `_build_export_df` path used by the Excel button for full parity. The minimal header-only fix is sufficient to stop the mislabeling.

### M1. Per-row hotkey fires keystrokes immediately with no countdown or abort window `[gui_ux]`
- **ID:** `gui-hotkey-no-countdown-abort`  |  **Fix regression risk:** low
- **Location:** trade_scanner_fh/hotkey.py:168-184 (send_ticker); dispatch with no countdown at trade_scanner_fh/gui/main_window.py:4308-4332 (_dispatch_hotkey_for_row); contrast bulk countdown/abort at trade_scanner_fh/tradestation.py:30 and 114-126; no coordinate-bounds validation in HotkeyConfig.has_position/normalized at trade_scanner_fh/hotkey.py:97-98 and 105-120
- **What:** Unlike the bulk bridge (configurable 1-30s countdown), send_ticker performs click → sleep(delay_ms) → typewrite → end-key with no countdown and no user abort window. The only delay is delay_ms (default 200ms, can be 0). For an explicitly affected ticker that yields nothing, remove its existing row or stamp next_earnings=NaT so a downstream > today consumer treats it as unknown. Build the keep mask from affected_tickers (when provided) rather than from new_df['ticker'].

### M25. Yahoo dates-bucket re-absorbs the reconciler's own compound output rows, laundering another source's value as Yahoo `[robustness]` [self-verified]
- **ID:** `robust-reconcile-yahoo-bucket-launders-source`  |  **Fix regression risk:** low
- **Location:** trade_scanner_fh/earnings_reconcile.py:126-144
- **What:** For source=='yahoo', _extract_dates_lookup includes any row whose source string contains 'yahoo' (L142, src_str.str.contains('yahoo')). The reconciler's own output labels are compound (e.g. 'nasdaq+yahoo_aug' = last_src=nasdaq, next_src=yahoo). Such a row's last_earnings was supplied by NASDAQ, but this bucket credits the whole row (both last and next) to the yahoo lookup, because it cannot tell which position yahoo supplied.
- **Impact:** On a re-run where the original nasdaq lookup no longer covers the ticker, _pick_last hands back nasdaq's stale last_earnings labeled as yahoo-derived — mislabeled provenance and potential propagation of a stale value through a step it didn't originate. In the common case nasdaq still outranks yahoo so the harm is masked; a latent edge-case wrong result.
- **Recommended fix:** Don't feed prior reconciler compound-output rows back into a single source's bucket as if both positions were that source's raw data: either only bucket rows whose source is exactly 'yahoo'/'legacy'/'unknown' (drop the contains() clause), or split so each position is credited to the source the label names. Persisting per-position source on the row would make this unambiguous.

### M26. Cached column widths truncate wider content on timeframe/period switch `[gui_ux]` [self-verified]
- **ID:** `gui-cached-column-widths-truncate`  |  **Fix regression risk:** low
- **Location:** trade_scanner_fh/gui/widgets.py:2476-2487
- **What:** The column-width fast path keys the width cache solely on the column-key tuple (tuple(c[1] for c in cols)), not on cell content. On the first render of a given column shape the widths are measured via resizeColumnsToContents() and snapshotted; every subsequent render of the same shape restores those exact pixel widths and skips re-measurement. Sequenced runs produce identical column shapes across periods but different value magnitudes (e.g. period A close=$3.41 vs period B close=$1483.22).
- **Impact:** When the user switches to a period whose values are wider than the first-rendered period's, columns stay at the narrower widths and content is visually truncated (e.g. '$1483...' clipped, large $M revenue clipped). The user sees apparently-incomplete/wrong numbers with no indication the cell is just too narrow.
- **Recommended fix:** Treat cached widths as minimums (grow via max(cached, freshly_measured) on each render, never shrink), or fold a coarse content-width signature into the cache key, or expose an auto-fit affordance. Lowest-regression: keep cached widths as minimums and let a header-divider double-click re-fit, or recompute only when a new max magnitude is detected.

### M27. Log panel QTextEdit grows unbounded in an always-on app / multi-hour fills `[efficiency]` [self-verified]
- **ID:** `eff-log-panel-unbounded-growth`  |  **Fix regression risk:** low
- **Location:** trade_scanner_fh/gui/widgets.py:2696-2742
- **What:** LogPanel.text is a plain QTextEdit with no setMaximumBlockCount cap. Every record routed through QtLogHandler is appended via _append and the document is never trimmed. The app is always-on and logging is verbose; bulk scrapes run for hours (Finviz ~11h, Finnhub ~9h, Zacks ~5-7h) emitting per-ticker progress + failures throughout.
- **Impact:** Over a long session / overnight bulk fill the document accumulates tens of thousands of blocks; memory grows monotonically and each append + scroll-to-bottom triggers a relayout of an ever-larger document, so UI responsiveness degrades over the run. The full log is already persisted to disk, so the in-memory copy serves no archival need.
- **Recommended fix:** Call self.text.document().setMaximumBlockCount(N) (e.g. 5000-10000) in __init__ so the widget self-trims oldest lines; the on-disk log retains full history. One-line addition with no behavioral change beyond bounding memory.
