# Update — 2026-06-18 (v5.3.0)

Small functionality addition: the per-row HOTKEY ticker sender's optional
**return click** now has its own adjustable wait time, independent of the
existing click → type delay. Version bumped `5.2.0 → 5.3.0`; exe rebuilt.

## Adjustable return-click delay

- **What:** a new **`Delay end-key → return click (ms)`** spinbox in
  `Settings → Hotkey Settings…` (range 0–5000 ms, default 200, 50 ms
  steps) — the pause between the end-sequence keystroke and the optional
  return click.
- **Why:** previously the return click reused the same `delay_ms` knob as
  the click → type delay, so the time the target app needs to *process the
  submit* before focus is moved back couldn't be tuned separately from the
  time the input field needs to *gain focus* before typing. They're
  different waits; now they're separate settings.
- **Backed by** a new `HotkeyConfig.return_delay_ms` field (clamped 0–5000
  in `normalized()`, like `delay_ms`); `send_ticker` sleeps on
  `return_delay_ms` before the return click instead of `delay_ms`.

## Backward compatibility

- A config saved before this field existed has no `hotkey/return_delay_ms`
  key. On load it **defaults to the saved `delay_ms`**, reproducing the old
  "shared delay" timing exactly — no silent behavior change for existing
  users. Fresh configs default both to 200 ms.
